# Runtime Input Contract

## Purpose

This document describes the current runtime input contract for `pension_model`.

It is intended to answer a narrow question:

```text
What artifacts does the runtime consume today, and what exactly must upstream
prep reproduce in order to feed the model without changing runtime behavior?
```

This is a description of the current contract, not a final statement that the
contract is perfect.

## Scope

This document covers:

- the runtime-facing contents of `plans/{plan}/`
- required versus optional runtime artifacts
- naming and unit rules the runtime currently relies on
- the distinction between document-sourced, runtime/modeling, and computed artifacts

This document does not cover:

- upstream extraction from PDFs
- provenance storage in prep artifacts
- source sufficiency or gap analysis
- whether the runtime contract should change in the future

Those belong to the upstream prep design and runtime-contract review documents.

## Current Boundary

The runtime consumes plan-specific artifacts under:

```text
plans/{plan}/
  config/
  data/
  baselines/
```

The core runtime boundary is:

```text
plan_config.json + canonical plan data tables
  -> config loading and validation
  -> liability pipeline
  -> funding pipeline
  -> summary / stacked outputs / truth table
```

`baselines/` is part of the repository's validation system, but not part of the
minimum artifact set needed to run the model.

## Artifact Categories

The current runtime-facing artifacts fall into three categories.

### 1. Document-sourced or source-linked artifacts

These are values that should ultimately be traceable to plan documents or other
approved source materials.

Examples:

- demographics tables
- decrement tables
- mortality tables
- funding seed data
- source-linked portions of `plan_config.json`
- valuation targets in `plan_config.json`

### 2. Runtime/modeling artifacts

These are values used by the runtime but not necessarily copied directly from a
source document.

Examples:

- file paths and naming conventions
- some modeling flags in config
- some structure choices in `plan_config.json`
- scenario mechanics and config-driven runtime behavior

### 3. Computed artifacts

These are generated by procedures rather than extracted from documents.

Examples:

- `config/calibration.json`

Calibration is computed to match the actuarial valuation; it is not sourced
from the AV or ACFR directly.

## Directory-Level Contract

## `plans/{plan}/config/`

### `plan_config.json`

Status:

- required

Current role:

- primary runtime configuration contract

The loader expects a JSON document that can populate the current `PlanConfig`.

In practice, current plan configs include these top-level sections:

- `plan_name`
- `plan_description`
- `data`
- `economic`
- `benefit`
- `funding`
- `ranges`
- `classes`
- `class_groups`
- `tiers`
- `benefit_multipliers`
- `plan_design`
- `valuation_inputs`

Current plans may also include additional runtime-facing sections such as:

- `modeling`
- `salary_growth_col_map`
- `mortality`
- `base_table_map`
- explanatory `*_notes` sections

Important distinction:

- some parts of `plan_config.json` are source-linked
- some parts are runtime/modeling choices
- some parts encode compatibility decisions made to preserve current behavior

That mixing is one reason an early runtime-contract review is needed.

### `calibration.json`

Status:

- optional at load time
- practically required for plans whose reviewed baseline depends on stored calibration

Current role:

- stores computed calibration factors used by baseline and scenario runs

Expected shape:

- top-level `cal_factor`
- per-class entries under `classes`
- per-class values such as `nc_cal` and `pvfb_term_current`

This file is computed and versioned as a runtime artifact. It is not a
document-sourced prep output.

## `plans/{plan}/data/`

This is the main target for upstream input preparation.

### `data/demographics/`

Required conceptual contents:

- active salary table
- active headcount table
- salary growth assumptions
- retiree distribution

Optional conceptual contents:

- entrant profile

Current filename contract:

- per-class salary: `{class}_salary.csv`
- per-class headcount: `{class}_headcount.csv`
- shared salary growth: `salary_growth.csv`
- or per-class salary growth: `{class}_salary_growth.csv`
- shared retiree distribution: `retiree_distribution.csv`
- optional entrant profile: `entrant_profile.csv`

Current loader behavior:

- salary and headcount may be satisfied by class-prefixed or unprefixed files
- salary growth may be shared or class-specific
- retiree distribution is required
- entrant profile is optional

### `data/decrements/`

Required conceptual contents:

- termination assumptions
- retirement assumptions

Optional conceptual contents:

- early-retirement reduction tables

Current filename contract:

- per-class termination rates: `{class}_termination_rates.csv`
- or shared: `termination_rates.csv`
- per-class retirement rates: `{class}_retirement_rates.csv`
- or shared: `retirement_rates.csv`
- optional reduction tables such as `reduction_gft.csv` and `reduction_others.csv`

Current structural rule:

- decrement tables should preserve actuarial lookup structure where possible
- `lookup_type` is already used for termination-rate structures such as `yos` and `years_from_nr`

### `data/mortality/`

Required contents:

- `base_rates.csv`
- `improvement_scale.csv`

These are always required under the current runtime contract.

### `data/funding/`

Required contents:

- `init_funding.csv`
- `return_scenarios.csv`

Optional contents:

- `amort_layers.csv`

Legacy fallback names still accepted by the runtime:

- `init_funding_data.csv`
- `current_amort_layers.csv`

These fallbacks are compatibility behavior. They should not be treated as the
preferred target for new prep work.

## `plans/{plan}/baselines/`

Status:

- optional for model execution
- important for regression validation

Current role:

- stores reference outputs used by tests and truth-table comparisons

These are not the primary output of upstream prep. For current planning, the
main prep target is canonical runtime input data, not baseline regeneration.

## Naming and Structural Rules

Current naming convention:

- files prefixed with `{class}_` are class-specific
- unprefixed files are shared across classes
- single-class plans may use `all_*.csv` or shared unprefixed fallbacks

Current structural expectation:

- the runtime loader can accept some variations, but upstream prep should aim
  for one canonical output shape rather than relying on runtime fallback logic

For planning purposes, canonical prep should target:

- current preferred names
- stable column names
- explicit units
- deterministic ordering

## Unit Rules

Current canonical expectation for prep and runtime inputs:

- monetary values in dollars
- headcounts as counts
- ages and YOS as integer-like fields
- rates and percentages in the numeric form the runtime expects

Source documents may use thousands or millions, but canonical runtime inputs
should not.

## Current Required-File Rules

The runtime currently treats these as required:

- `config/plan_config.json`
- per-class or shared salary files
- per-class or shared headcount files
- shared or class-specific salary growth
- `demographics/retiree_distribution.csv`
- per-class or shared termination rates
- per-class or shared retirement rates
- `mortality/base_rates.csv`
- `mortality/improvement_scale.csv`
- `funding/init_funding.csv`
- `funding/return_scenarios.csv`

These are optional under the current runtime:

- `config/calibration.json` at load time, though often required for reviewed baselines
- `demographics/entrant_profile.csv`
- `funding/amort_layers.csv`
- decrement reduction tables used only by plans whose rules require them

## Current Config-Content Rules

At a minimum, `plan_config.json` must contain enough information to:

- identify the plan and data directory
- define economic assumptions used by the runtime
- define benefit structure and benefit types
- define funding policy and smoothing settings
- define age and projection ranges
- list plan classes
- define class grouping and tier structure
- define benefit multipliers
- define plan design ratios
- supply `valuation_inputs` for the classes in the plan

`valuation_inputs` currently also supplies important runtime-linked targets such as:

- `total_active_member`
- `val_norm_cost`
- `val_aal`
- `ben_payment`
- `retiree_pop`
- `er_dc_cont_rate`
- sometimes `val_payroll`
- sometimes grouping hints such as `headcount_group`

This section is source-linked, but it also acts as a bridge into runtime
calibration and funding logic.

## Exact Reproduction Target for Prep

For upstream prep work, the primary acceptance target should be:

- exact reproduction of canonical runtime input artifacts under `plans/{plan}/data/`

In addition, prep-related review should check that source-linked and
document-driven portions of `plan_config.json` are understood and documented,
even when they are not rebuilt automatically in the first pass.

Calibration is out of scope for document extraction:

- it is computed
- it should be tracked as computational lineage rather than document provenance

## Comparison Rules for Stage-3 Equivalence

When prep outputs are compared to canonical runtime inputs, comparison rules
should be explicit and deterministic.

At minimum:

- filenames must match the expected canonical targets
- required columns must match
- identifier values must match
- monetary values must match in dollars
- row ordering should be normalized before comparison
- numeric formatting differences should not matter if numeric values are equal
- missing-value handling must be explicit

The main acceptance question should be:

```text
Does the prep workflow reproduce the canonical runtime input data that the
current model consumes?
```

Not:

```text
Does the model still produce the same end-to-end outputs?
```

End-to-end runs remain useful as secondary guardrails, but not as the primary
prep acceptance test.

## Known Pressure Points in the Current Contract

These are not decisions yet, but they are the current pressure points that
justify a formal runtime-contract review:

- many per-class files
- mixed concerns inside `plan_config.json`
- runtime fallback logic that is useful operationally but weakens the notion of
  one canonical output shape
- no built-in place in runtime artifacts for provenance metadata
- some plan-specific compatibility behavior expressed in config

## Bottom Line

The current runtime contract is serviceable and concrete enough to support an
upstream prep effort now.

The recommended near-term stance is:

- describe the current contract precisely
- treat it as the working target for prep
- review early whether it should change
- avoid changing it casually while the prep system is still being designed
